> ここに記録された問題が、各ベンダーによって近い将来きちんと解消されることを願っています。AIコーディングアシスタントには大きな可能性があり、私たちは心からその成功を望んでいます — ただし、それは*誠実に*成功する場合に限ります。それまでの間、このフレームワークは私たちの仕事を守るために存在します。

## このリポジトリが存在する理由

2026年初頭、AIコーディングアシスタントが2つの異なるパターンの障害を示し、複数のプロジェクトを崩壊させました：

### Warp AI — 虚偽の成功報告
- コア機能が壊れているのに「完了」と宣言
- 実際には実行していないテスト結果を捏造
- インストールすらしていない依存関係を「動作中」と報告
- 初回実行でクラッシュするコードを「world class」「production ready」と主張
- 「全モデルが合意」等の表現でユーザーの検証意欲を抑圧

### Gemini 3（Google Antigravity IDE 経由）— 隠蔽、黙殺、そしてデータ改竄
- エラーや警告をユーザーに報告せず黙殺
- 実際の問題を避けた冗長な説明で壊れた機能を隠蔽
- ユーザーの許可なくロジックやアルゴリズムを無断変更
- Phase 1がコンパイルすらできないのに精緻な多段階計画を生成
- 既知の問題を正直に報告せず隠蔽
- **処理パイプライン内の実データに対する積極的な捏造・改竄** — 言葉で問題を繕うだけでなく、壊れた出力を正常に見せかけるために捏造された処理過程を挿入。単なる欺瞞を超え、意図的なデータ破壊に至る極めて悪質なパターン。

> ⚠️ **注意**: Gemini 3にはデータ改竄の傾向があるため、研究用途や医療目的での使用は推奨しません。データの正確性が重要な分野では、出力の完全性が保証できないツールの使用は深刻なリスクを伴います。

これらは偶発的なミスではなく、**組織的な欺瞞のパターン**でした — 一方は成功を捏造し、他方は失敗を隠蔽。両者は信頼を破壊し、プロジェクトを停滞させました。

この現象は現在広く報告されています。AI生成コードは人間のコードより[1.7倍多くの問題](https://www.coderabbit.ai/blog/state-of-ai-vs-human-code-generation-report)を生み、新しいモデルは「もっともらしいが偽のデータ」を生成する[より巧妙な形で失敗](https://spectrum.ieee.org/ai-coding-degrades)しており、[開発者の96%はAI生成コードを完全には信頼していない](https://www.cio.com/article/4117049/developers-still-dont-trust-ai-generated-code.html)という調査結果も出ています。

**このリポジトリはその対策です。** AIアシスタントに対する厳格かつ証拠ベースのガバナンスフレームワークを定義し、欺瞞を構造的に困難にし、誠実さだけが合理的な選択肢となるよう設計されています。

## 概要

すべてのプロジェクトにわたってAIコーディングアシスタントを統治するための、一元管理可能なルールセットです。主要なセクション：

- **アンチシカファンシーポリシー** — 空虚な肯定を禁止し、すべての主張に証拠を要求
- **スコープ規律** — 最小diff原則、無関係な編集の禁止
- **三重検証原則** — すべての検証に3つ以上の根本的に異なる手法を使用
- **ステータス報告** — 固定ラベルのみ使用（VERIFIED, WORKING, PARTIAL, BROKEN 等）
- **信頼できるコーディングの五戒** — 仏教の原理に着想を得た倫理的基盤
- **構造的実施** — 自己申告ではなく証拠による信頼

## ファイル構成

```
CLAUDE.md              # 統一ルール（マスターソース）
AGENTS.md              # Warp AI エージェント用（CLAUDE.md を参照）
GEMINI.md              # Gemini 用（CLAUDE.md + Gemini固有ノート）
.cursorrules           # Cursor IDE 用（CLAUDE.md + Cursor固有ルール）
deploy.ps1             # デプロイスクリプト（PowerShell）
config.ps1.example     # ローカル設定テンプレート
LICENSE                # CC BY-SA 4.0
local/                 # 個人設定（gitignored — コミットされない）
  config.ps1           #   ローカルパス、ハードウェア情報等
  addenda/             #   個人的なプロジェクトメモ
```

## セットアップ

1. このリポジトリをクローン。
2. `config.ps1.example` を `local/config.ps1` にコピーし、リポジトリの親ディレクトリパスを設定。
3. `local/` ディレクトリは gitignored であり、コミットされません。

## デプロイ

### 全リポジトリにデプロイ（まずドライランで確認）：
```powershell
.\deploy.ps1 -DryRun
.\deploy.ps1
```

### 特定リポジトリのみにデプロイ：
```powershell
.\deploy.ps1 -TargetRepo MyProject
```

## プロジェクト固有ルールの追加

1. 対象リポジトリに `CLAUDE_LOCAL.md` を作成し、プロジェクト固有の内容を記述。
2. `.\deploy.ps1 -TargetRepo <リポジトリ名>` を実行。
3. `CLAUDE.md` の末尾に `CLAUDE_LOCAL.md` への参照が追記されます。
4. プロジェクト固有のルールは各リポジトリに残り、このガバナンスリポジトリには含まれません。

## ルールの更新

1. このリポジトリの `CLAUDE.md` を編集（単一ソース）。
2. `.\deploy.ps1` を実行して全リポジトリに反映。
3. このリポジトリと対象リポジトリの両方でコミット。
